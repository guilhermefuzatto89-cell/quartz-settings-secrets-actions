# 1.2.1 — Google Doc (Documentação Oficial da Pesquisa)

> [!info] Sobre esta nota
> Síntese estruturada da documentação oficial da **Central da Pesquisa Google** (`developers.google.com/search`). Diferente das notas de mercado ([[1.3 - SEO]]), aqui o que vale é o que o **próprio Google documenta** — sem intermediário. Aplicável a [[2.1 - Site]], [[2.1.2 - E-commerce]], [[2.2 - Marketplace]].

Fonte primária: [developers.google.com/search](https://developers.google.com/search?hl=pt-br)

## Os 3 estágios da Pesquisa Google

O Google opera em **três estágios sequenciais** — nem toda página passa pelos três:

```
1. RASTREAMENTO → 2. INDEXAÇÃO → 3. EXIBIÇÃO DOS RESULTADOS
```

### Estágio 1 — Rastreamento (Crawling)

O **Googlebot** visita páginas para download de texto, imagens e vídeos. Descoberta de URLs por três caminhos:

- Links em páginas já conhecidas pelo Google
- **Sitemaps XML** submetidos pelo dono do site
- Envio manual via Search Console

O Googlebot **renderiza JavaScript** com versão recente do Chrome — conteúdo JS é rastreado, mas com atraso em relação ao HTML estático. Para sites JS-heavy, SSR/SSG é muito mais seguro.

> [!warning] O Google não garante rastreamento
> Mesmo seguindo todas as diretrizes, o Google **não garante** rastrear, indexar ou exibir a página. Não existe pagamento para rastrear com mais frequência ou rankear mais alto.

**Problemas comuns:** erros de servidor (HTTP 500), problemas de rede/DNS, regras `robots.txt` bloqueando acesso indevidamente.

### Estágio 2 — Indexação (Indexing)

Após rastrear, o Google analisa o conteúdo e determina qual URL é **canônica** (a que representa o cluster de páginas similares). A canônica é a que aparece nos resultados. O índice armazena: idioma, país, usabilidade e outros indicadores.

**Problemas comuns:** baixa qualidade de conteúdo, regras `noindex`, design JS sem renderização server-side.

### Estágio 3 — Exibição dos Resultados (Serving)

O Google considera **centenas de fatores**: localização, idioma, dispositivo, histórico e intenção da consulta. Os recursos de SERP variam conforme a query — "oficina de bicicleta" mostra resultados locais; "bicicleta moderna" mostra imagens.

## Fundamentos técnicos obrigatórios

### Requisitos para rastrear e indexar

- Página acessível ao Googlebot (não bloqueada por `robots.txt` ou firewall)
- Resposta **HTTP 200**
- Conteúdo **renderizável** (HTML puro ou JS que o bot consegue processar)
- **HTTPS** (sinal de confiança, não requisito absoluto)
- **Mobile-friendly** — desde 2021 o Google usa **indexação mobile-first**: a versão mobile é a base para indexação e ranking em todos os dispositivos

### Políticas de spam (o que derruba)

- Conteúdo gerado automaticamente **sem valor** para o usuário
- **Cloaking** — mostrar conteúdo diferente ao bot vs. ao usuário
- Links comprados ou esquemas de manipulação de PageRank
- Texto/links ocultos (cor igual ao fundo, tamanho 0)
- Redirecionamento enganoso
- Conteúdo scrapeado sem valor agregado
- Spam gerado por usuários em fóruns/comentários sem moderação

## Controles de rastreamento e indexação

### robots.txt

Arquivo em `/robots.txt` que instrui **quais áreas o Googlebot pode rastrear**. Controla rastreamento, **não indexação**.

```
User-agent: *
Disallow: /admin/
Disallow: /checkout/
Allow: /produtos/
```

> [!warning] Confusão clássica — robots.txt vs noindex
> `robots.txt` bloqueia **rastreamento**. Para bloquear **indexação**, use `<meta name="robots" content="noindex">` ou `X-Robots-Tag: noindex` no header HTTP.
> **Armadilha:** bloquear via `robots.txt` uma página que você quer fora do índice pode resultar nela aparecendo **no índice sem conteúdo**, porque o Google indexa URLs linkadas mesmo sem ter rastreado o conteúdo delas.

### Metatag Robots

```html
<meta name="robots" content="noindex">           <!-- Bloquear indexação -->
<meta name="robots" content="nofollow">          <!-- Não seguir links -->
<meta name="robots" content="nosnippet">         <!-- Sem snippet na SERP -->
<meta name="robots" content="noindex, nofollow"> <!-- Combinado -->
<meta name="robots" content="max-snippet:160">   <!-- Limitar tamanho do snippet -->
```

### Canonização (rel="canonical")

Indica ao Google qual URL é preferida quando há conteúdo duplicado:

```html
<link rel="canonical" href="https://example.com/produto-x/" />
```

**Quando usar:** URLs com parâmetros (`?cor=azul&tamanho=M`), versões HTTP/HTTPS duplicadas, paginação, versões mobile/desktop separadas, conteúdo sindicado.

> [!tip] Canonical é sugestão, não ordem
> O Google pode **ignorar** o canonical se achar que outra URL é mais adequada. É sinal forte, não diretiva absoluta como `noindex`.

## Sitemaps XML

Lista de URLs para facilitar descoberta. Útil especialmente para: sites grandes (> 1.000 páginas), sites novos (poucos links externos), páginas órfãs (sem links internos apontando pra elas), páginas de mídia.

**Tipos suportados pelo Google:**
- **Padrão** — páginas HTML
- **Imagens** — `<image:image>` (garante indexação de imagens)
- **Vídeos** — `<video:video>` (título, duração, thumbnail na SERP)
- **Google Notícias** — para publishers credenciados

**Limite:** 50.000 URLs e 50 MB por arquivo. Sites maiores usam **sitemap index** que aponta para múltiplos sitemaps filhos.

## Redirecionamentos

| Código | Tipo | Passa autoridade? | Quando usar |
|---|---|---|---|
| **301** | Permanente | ~100% do PageRank | Mudança definitiva de URL |
| **302** | Temporário | Não | Redirecionamento provisório |
| **308** | Permanente (preserva método) | ~100% | POST requests permanentes |
| **404** | Não encontrado | Perde autoridade | Página removida sem substituto |
| **410** | Gone | Perde autoridade | Sinaliza remoção intencional (mais forte que 404) |

> [!danger] Cadeia de redirecionamentos
> Mais de 3 redirects em cadeia pode fazer o Googlebot abandonar o rastreamento. Mantenha direto: A → C, nunca A → B → C → D.

## SEO em JavaScript — o que a documentação diz

O Google **executa JavaScript**, mas há nuances críticas:

1. **Renderização em duas ondas:** HTML primeiro, JS depois (pode levar dias/semanas para páginas novas)
2. **Conteúdo lazy-loaded:** deve ser visível após scroll simulado pelo bot — caso contrário, não é indexado
3. **Renderização dinâmica:** serve HTML estático ao bot e JS ao usuário — solução de contorno aceita pelo Google, mas SSR/SSG é preferido
4. **CSR puro (React/Vue sem SSR):** alto risco — o Google pode indexar página vazia se o JS falhar ou demorar

**Recomendação oficial:** use **Server-Side Rendering (SSR)** ou **Static Site Generation (SSG)** para todo conteúdo que você quer indexado. Ver implementação em [[2.1 - Site]] e [[2.3 - App]].

## Recursos de SERP disponíveis (Galeria de Elementos)

O Google documenta os seguintes elementos que podem aparecer nos resultados:

- **Resultado orgânico padrão** — título, URL, snippet
- **Resultado de imagem** e **vídeo**
- **Featured Snippet** — caixa de resposta direta no topo (posição zero)
- **AI Overviews** — resumo gerado por IA (lançado 2024, expandido em 2025-2026)
- **Resultados locais / Google Maps** — negócios com localização física
- **Sitelinks** — links adicionais abaixo do resultado principal para sites de marca
- **Rich Results** — via dados estruturados (estrelas, preço, FAQ expandido, etc.)
- **Google Discover** — feed personalizado sem pesquisa explícita
- **Painel de conhecimento (Knowledge Panel)** — marcas e entidades conhecidas
- **Carrossel de resultados** — produtos, receitas, filmes

### Title Link

O `<title>` da página é o principal input para o título do resultado. O Google pode **reescrever** se não descrever bem a página. Práticas recomendadas:

- Título único por página, descritivo e conciso (< 60 caracteres)
- Inclua a marca (geralmente no final: "Produto X | Marca")
- Evite genéricos ("Página inicial", "Novo") ou keyword stuffing

### Snippets e Featured Snippets

O snippet vem de `<meta name="description">` ou de trechos da página que o Google julga mais relevantes para a query. Featured Snippets aparecem em queries interrogativas ("como fazer", "o que é", "qual é") — **não são solicitáveis**. Para otimizar: resposta direta no início, H2/H3 para perguntas, listas para passos.

## Dados Estruturados (Schema Markup)

Anotações em **JSON-LD** dentro do `<script>` que ajudam o Google a entender o tipo de conteúdo e podem ativar **Rich Results**.

### Formato recomendado (JSON-LD)

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Título do artigo",
  "author": { "@type": "Person", "name": "Nome do Autor" },
  "datePublished": "2026-05-23",
  "dateModified": "2026-05-23"
}
</script>
```

> [!tip] JSON-LD é o formato preferido pelo Google
> Diferente de Microdata e RDFa (que ficam no corpo do HTML), JSON-LD é separado — mais fácil de manter e atualizar sem alterar o HTML visível.

## Galeria completa de tipos de Schema documentados

### Conteúdo editorial
| Schema | O que ativa na SERP |
|---|---|
| `Article` | Artigo de destaque no Google Notícias e Discover |
| `FAQPage` | FAQ expandido diretamente na SERP |
| `HowTo` | Passos enumerados no resultado |
| `Recipe` | Foto, tempo de preparo, ingredientes, avaliação |
| `Dataset` | Painel de conjunto de dados |
| `DiscussionForum` / `QAPage` | Resultado de fórum/Q&A com contexto de discussão |

### E-commerce / Negócios (essencial para [[2.1.2 - E-commerce]])
| Schema | O que ativa na SERP |
|---|---|
| `Product` + `Offer` | Preço, disponibilidade, estrelas de avaliação |
| `MerchantListing` | Integração com Google Shopping |
| `ProductVariants` | Variações (cor, tamanho) visíveis na SERP |
| `LocalBusiness` | Endereço, horário, avaliações em resultados locais |
| `Organization` | Knowledge Panel da empresa |
| `LoyaltyProgram` | Benefícios do programa de fidelidade |
| `ReturnPolicy` / `ShippingPolicy` | Políticas visíveis no resultado de produto |

### Mídia
| Schema | O que ativa na SERP |
|---|---|
| `VideoObject` | Thumbnail, duração, timestamp no resultado de vídeo |
| `ImageObject` | Metadados de licença e autoria de imagem |
| `WebStory` | Carrossel de Web Stories |

### Outros
| Schema | O que ativa na SERP |
|---|---|
| `Event` | Data, local, status, link de ingressos |
| `JobPosting` | Vaga no Google Jobs com salário e empresa |
| `Course` | Curso com provedor e descrição |
| `BreadcrumbList` | Navegação estrutural visível na URL do resultado |
| `SoftwareApplication` | Avaliação, preço e sistema operacional do app |
| `ReviewSnippet` | Estrelas de avaliação agregada |

> [!tip] Ferramenta oficial: Rich Results Test
> Valide seu schema em tempo real: [search.google.com/test/rich-results](https://search.google.com/test/rich-results)

> [!warning] Schema correto não garante Rich Result
> O Google decide se exibe mesmo com schema válido. Conteúdo de baixa qualidade ou que viola políticas não recebe Rich Results.

## AI Features — o novo capítulo da documentação (2025-2026)

O Google adicionou à documentação oficial em 2025-2026:
- **"AI Features"** — seção em Classificação e aspecto da pesquisa
- **"Optimizing for generative AI search"** — novo guia dedicado nos fundamentos
- **"Guidance on using generative AI"** — orientações sobre conteúdo gerado por IA

### O que o Google documenta sobre IA generativa

O Google **não proíbe** conteúdo gerado por IA. O critério oficial é **qualidade e utilidade**, não a origem. No entanto:

- Conteúdo gerado em massa para manipular rankings **viola políticas de spam**
- O foco deve ser em conteúdo "**útil, confiável e que prioriza as pessoas**"
- **E-E-A-T** (Experience, Expertise, Authoritativeness, Trustworthiness) continua sendo o framework central de avaliação
- Conteúdo gerado por IA **sem supervisão humana ou valor real** = spam documentado

### Para aparecer em AI Overviews (posição oficial)

A documentação indica que o Google usa os **mesmos sistemas de qualidade do Search** para decidir quais fontes citar em AI Overviews. Conclusão direta: **melhorar o SEO tradicional melhora as chances de citação em AI Overviews** — não são canais separados. Aprofundamento em [[1.3 - SEO]].

## E-commerce SEO — guia específico da documentação

Aplicável a [[2.1.2 - E-commerce]], [[2.2 - Marketplace]], [[2.2.1 - Amazon]], [[2.2.3 - Mercado Livre]], [[2.2.4 - Shein]].

### Onde produtos podem aparecer no Google

- Resultados orgânicos via SEO + schema `Product`
- Google Shopping via Merchant Center + dados estruturados
- Google Imagens via `alt text` + schema `ImageObject`
- Knowledge Panel de produtos
- Featured Snippets de produto

### Stack de schema indispensável para e-commerce

```html
<!-- Exemplo mínimo para produto com avaliação -->
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Product",
  "name": "Nome do Produto",
  "offers": {
    "@type": "Offer",
    "price": "99.90",
    "priceCurrency": "BRL",
    "availability": "https://schema.org/InStock"
  },
  "aggregateRating": {
    "@type": "AggregateRating",
    "ratingValue": "4.8",
    "reviewCount": "127"
  }
}
</script>
```

### Estrutura de URLs para e-commerce (recomendação oficial)

- ✅ `/categoria/produto-nome/`
- ✅ `/camisetas/camiseta-branca-algodao/`
- ❌ `/produto?id=12345&color=azul&ref=homepage`

### Paginação e carregamento incremental

- **Paginação tradicional** com `<a href>` crawláveis = melhor opção para SEO
- **"Ver mais"** deve atualizar a URL (não apenas AJAX invisível)
- **Infinite scroll** sem âncoras de URL = o Googlebot **não rastreia** o que não tem URL própria

## Search Console — ferramenta oficial de monitoramento

| Relatório | O que mostra | Para que usar |
|---|---|---|
| **Desempenho** | Cliques, Impressões, CTR, Posição média | Identificar oportunidades de title/meta description |
| **Cobertura de índice** | Páginas indexadas, erros, excluídas | Diagnosticar problemas de indexação |
| **Core Web Vitals** | LCP, INP, CLS por grupo de URLs | Priorizar melhorias técnicas |
| **Rich Results** | Quais rich results estão ativos/com erro | Validar e debugar schema markup |
| **Links** | Backlinks externos e links internos | Auditar perfil de autoridade |
| **Inspeção de URL** | Status de rastreamento/indexação por URL | Debug de página específica |
| **Sitemaps** | Status de submissão e erros | Validar cobertura de indexação |

### Operadores de pesquisa para debug

- `site:example.com` — todas as URLs indexadas do domínio
- `site:example.com/categoria/` — páginas indexadas de uma subpasta específica
- `site:example.com "texto-único"` — verificar se uma página específica está no índice

## Sistemas de classificação documentados pelo Google

O Google documenta oficialmente que usa **múltiplos sistemas** que trabalham em conjunto, não um único algoritmo. Os mais relevantes:

- **Sistema de conteúdo útil** — avalia se o conteúdo prioriza o usuário ou foi criado para SEO
- **Sistema de avaliações** — avalia qualidade de reviews e análises de produtos
- **RankBrain** — ML para interpretar queries ambíguas ou novas
- **BERT / MUM** — modelos de linguagem para compreender contexto e nuances de busca
- **PageRank** — autoridade de links (ainda presente, não é o único sinal)
- **SpamBrain** — detecta manipulação algorítmica de links e conteúdo

> [!note] "Ranking factors" não são fixos
> O Google deixa claro que seus sistemas variam por tipo de query, dispositivo, localização e contexto. Não existe lista única e imutável de "200 fatores" — é um ecossistema dinâmico.
